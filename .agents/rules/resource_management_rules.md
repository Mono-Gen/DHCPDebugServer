# リソース管理ルール (Resource & Network Management)

長時間稼働・常時接続型プログラムにおいて、システムリソースとネットワークリソースの適切な管理を義務付けます。

## 第1章：システムリソース管理

### 1. ファイル・ソケット・接続の確実なクローズ
ファイル・ソケット・DB接続は `with` 文またはtry/finallyで管理し、例外発生時でも必ずクローズされることを保証してください。

### 2. 無制限に増え続けるデータ構造の禁止
バッファ・ログリスト・キャッシュ・キューには必ず上限サイズを設定すること。無制限に追加し続けるとメモリが肥大化し、長時間稼働後にクラッシュします。

### 3. スレッド・タイマーの確実な停止
終了時（Ctrl+C・シャットダウン等）にバックグラウンドスレッドやタイマーが確実に停止されるよう、シグナルハンドラーや停止フラグを実装すること。停止しないとゾンビスレッドやポートの競合が発生します。

### 4. 長時間稼働での動作確認の義務
長時間稼働を想定したプログラムは短時間テストだけで完了とせず、時間経過でメモリ使用量・スレッド数・接続数が増え続けていないか確認すること。

## 第2章：ネットワークリソース管理

### 5. ソケットのタイムアウト設定の義務
全てのソケットに必ずタイムアウトを設定すること。タイムアウトなしのブロッキング受信は、応答がない場合にプログラムが永久に停止するため厳禁です。

### 6. マルチキャストグループのJoin/Leave管理
マルチキャストグループへの参加（Join）と離脱（Leave）は必ずセットで実装すること。
- **Join時**: 使用するネットワークインターフェース（NIC）を明示的に指定すること。デフォルトNICへの参加はマルチホーム環境で意図しない動作を引き起こします。
- **Leave時**: プログラム終了・ストリーム停止・再接続時に必ずLeaveを実行すること。Leaveなしで終了するとネットワークスイッチが不要なトラフィックを送り続け、他機器のネットワーク品質に影響を与えます。
- **再接続時**: Leave → 待機 → Join の順序を守り、旧グループメンバーシップが残らないようにすること。

### 7. ネットワークインターフェースの明示的な指定
ソケットのバインドやマルチキャストの送受信では、`0.0.0.0`（全NIC）への依存を避け、対象のNICのIPを明示的に指定してください。複数のNICが存在する環境（有線＋無線等）では意図しないNICが使用されることがあります。

### 8. UDPパケットロスの考慮
UDPを使用する場合、パケットロスが発生することを前提に設計してください。
- 受信が途絶えた場合のタイムアウト処理を必ず実装すること。
- 重要なデータの送信はTCPを使用するか、アプリケーション層で再送機構を実装すること（`device_control_rules.md` 参照）。
