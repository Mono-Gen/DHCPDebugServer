# コードスタイルガイドおよび開発ルール

## マニュアルの作成
- 英文 (`manual_EN.md`) および 日本語 (`manual_JP.md`) のマニュアルを、プロジェクトのルートディレクトリ（カレントディレクトリ）に作成・更新すること。
- **ドキュメントの同期**: 機能変更や使い方の更新を行う際は、両言語のマニュアルを同時に更新し、内容に矛盾がないようにすること。

## セキュリティと品質
- 安全第一、セキュリティチェック、バリデーションチェック、およびコードのリファクタリングを必ず行うこと。
- セキュリティ全般に関するルール（絶対パスの禁止・機密情報のハードコード禁止・環境変数の利用・公開前スキャン等）は `global_rules.md` を参照すること。
- コンパイルする前に、少なくとも2回以上はコードに問題がないか、仕様を満たしているかをチェックすること。
- 回答や修正の速度よりも安定性やバグがないことを優先すること。

## 開発ルール
- **依存関係の管理**: 新しいライブラリをインストールした際は、必ず `requirements.txt` を最新の状態に更新すること。
- **仮想環境の使用（必須）**: PCのグローバル環境を汚染しないため、プロジェクトの実行・ライブラリ管理には必ず言語に応じた仮想環境を使用すること。
  - Python: `venv` または `virtualenv`
  - Node.js: `nvm` + プロジェクトごとの `node_modules`
  - Ruby: `rbenv` / `bundler`
  - その他: 対象言語の標準的な仮想環境ツールを使用すること。


## 効率的な開発プロセス
- **タスクの進捗管理**: 常に `task.md` を最新の状態に保ち、現在のステップを明確にすること。大きなタスクは小さなサブタスクに分割して実行すること。
- **段階的な実装と検証**: 一度に多くの変更を行わず、機能単位で実装・検証を繰り返すこと。変更のたびに、関連する機能が破壊されていないか確認すること。
- **コードのモジュール化**: 1つの関数やクラスは1つの責任を持つように設計し、必要に応じてファイルを分割して再利用性を高めること。
- **テストの作成**: 値の変換・計算・パース処理など、実機不要で検証できるロジックには必ずユニットテストを作成すること。
- **テストファイルの配置**: テストファイルは `tests/` フォルダに配置し、対象ファイル名の先頭に `test_` を付けた名前にすること（例: `utils.py` → `tests/test_utils.py`）。
- **実機依存テストの省略**: 実機への接続が必要な処理のテストは省略可とするが、その場合はコメントで「実機テスト必須」と明記すること。
- **診断スクリプトの除外**: `scratch/` や `diagnostics/` 内のスクリプトはテスト対象外とする。

# GitHub



## バージョン管理
- バージョン番号は `vX.Y.Z` 形式（Semantic Versioning）で管理すること。
  - `X`（メジャー）: 大きな変更・互換性が壊れる変更
  - `Y`（マイナー）: 後方互換の機能追加
  - `Z`（パッチ）: バグ修正のみ
- 開発中・テストビルドには `-alpha`（荒削り）または `-beta`（ほぼ完成のテスト）のサフィックスをつけること（例: `v1.1.0-beta`）。
- サフィックスなし（例: `v1.0.0`）が正式リリースであり、他者に配布できる状態とみなすこと。

## リリース前チェック
- 正式リリース（サフィックスなし）の前に、以下を必ず確認すること。
  - 絶対パス（`C:\Users\...` 等）がコード・ドキュメントに残っていないか全件検索する。
  - APIキー・パスワード・トークン等の機密情報がハードコードされていないか確認する。
  - `.env` ファイルが `.gitignore` に含まれているか確認する。
