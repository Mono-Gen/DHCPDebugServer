# UI/UXデザインルール (UI/UX Design Rules)

機器制御パネル、Stream Deckプラグイン、ネットワーク監視ツール等のUIを実装する際は、以下の原則に従ってください。

## 第1章：状態の視覚的表現（最重要）

### 1. 状態の4段階を必ず区別する
ユーザーが機器の本当の状態を誤認しないよう、以下の4つの状態を視覚的に明確に区別してください。
- **確定済み（Confirmed）**: Tallyや状態確認で検証済みの状態。通常の色・表示で示す。
- **処理中/未確認（Pending）**: コマンド送信後、Tally待ちの状態。グレーアウト、スピナー、または点滅で示す。
- **エラー（Error）**: 操作が失敗した状態。赤色または警告アイコンで示す。
- **オフライン（Offline）**: 機器との通信が切断されている状態。単なるOFF状態と区別し、専用の表示（例: 斜線アイコン、グレー）を使用すること。

### 2. カラーパレットの統一
プロジェクト内で色の意味を統一し、画面・コンポーネント間で意味が変わらないようにしてください。
- 🟢 **緑**: アクティブ / 正常 / ON / 確定済み
- 🔴 **赤**: エラー / 機器オフライン / 危険操作
- 🟡 **黄**: 警告 / 処理中 / 遷移中
- ⚫ **グレー**: 未接続 / 未確認 / 無効

### 3. リアルタイム更新のフィードバック
自動更新（ポーリング・サブスクライブ）を実装している場合、データが最新であることをユーザーが確認できるようにしてください。
- 「最終更新時刻」または「接続中インジケーター（ドット点滅等）」を表示すること。
- 更新が途絶えた（タイムアウト）場合は、データが古い可能性があることを視覚的に示すこと。

## 第2章：機器制御UIの安全設計

### 4. 危険操作への確認ステップ
取り消しが困難な操作（電源OFF、信号切替、初期化等）には確認ダイアログを実装してください。
- **原則**: 確認ダイアログを表示し、意図しない操作を防ぐこと。
- **例外**: ライブ現場など、即時操作が求められる環境では確認ステップが逆効果になる場合がある。設定でOFF可能にするオプションを検討すること。

### 5. UIのロールバック表示
`device_control_rules.md` の「エラー処理とUIのロールバック」と連動し、Tallyエラー・タイムアウト時にはUIを「最後に確認できた正しい状態」に戻し、ユーザーに操作が失敗したことを明確に通知してください。

## 第3章：Stream Deck・専用コントローラー向けルール

### 6. アイコン設計
- アイコンは **72×72px** を基準とし、視認性を優先してデザインすること。
- 状態は**色**と**テキスト**の両方で表現し、色覚特性を持つユーザーでも識別できるようにすること。
- 状態テキストは **最大4〜6文字** に収め、小さな画面でも判読できるフォントサイズを使用すること。

### 7. ボタンの状態フィードバック
- ボタン押下後、即座に「処理中」状態（Pending）を表示し、ユーザーに操作が受け付けられたことを伝えること。
- Tallyを受信するまでは確定表示に切り替えないこと。

## 第4章：エラーと通知

### 8. エラーメッセージの書き方
ユーザーに対して技術的な内部コードだけを表示しないでください。エラーメッセージには以下の3点を含めること。
- **何が起きたか**: 例「機器との通信が切断されました」
- **なぜ起きたか（可能であれば）**: 例「タイムアウト（5秒）」
- **どうすればよいか**: 例「ネットワーク接続を確認し、再接続してください」

### 9. 通知の優先度
ユーザーの作業を妨げないよう、通知の重要度に応じて表示方法を選択してください。
- **高（即時対応が必要）**: モーダルダイアログ
- **中（注意が必要）**: 画面端のトースト通知
- **低（情報提供）**: ステータスバーへの表示
