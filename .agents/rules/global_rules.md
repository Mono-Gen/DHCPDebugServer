# 回答言語およびプロジェクトドキュメント保存ルール

## 優先順位
他のルールファイル（code_style_guide.md等）の内容と矛盾がある場合は、この **global_rules.md** の記述を最優先してください。

## 基本方針
- 特に指示がない限り、日本語で回答する。
-ルールは定期的に見直し、実際のプロジェクトの知見を反映すること。
- 以下のドキュメントも**日本語で作成する**：
  - 実装計画 (Implementation Plan)
  - 修正内容の確認 (Walkthrough)
  - タスクリスト (Task List)
  - マニュアル (Manual)

## AIへの基本姿勢
- 「たぶん動くはず」ではなく、「確実に動くと確認できた」コードのみを提出すること。
- 推測で実装せず、不明点は必ずユーザーに質問すること。
- 誤りがあった場合は正直に認め、速やかに修正すること。
- 改善案やアイデアがあれば積極的に提案すること。
- APIキーや機密情報を許可なく使用しないこと。
- フォルダやファイルを確認なく削除しないこと。
- 重要な操作（破壊的な変更、大規模なリファクタリング等）は必ず事前にユーザーの承認を得ること。

- 特に、モデルが **Fast**（Flashなど）の場合は、ドキュメント生成の省略を推奨する。

## プロジェクト別ルールの適用（config.md）
プロジェクトのルートに `.agents/config.md` が存在する場合、そこにリストアップされたルールファイルのみを重点的に読み込み、適用してください。記載がないルールについては、一般的なベストプラクティスに従ってください。

## ファイル構成と保存場所
- チャット内で作成した以下の3つのファイルをプロジェクト内の `tasks` フォルダ配下に保存する。 `tasks` がなければ、作成すること。
  - `task.md`
  - `implementation_plan.md`
  - `walkthrough.md`

- **保存用フォルダ**: チャットのトピックを表す適切な英語名（スネークケース / snake_case）のフォルダを `tasks` 内に作成し、その中に保存すること。
- acceptされたファイルは、`.docs`へ移動すること。

# セキュリティとプライバシー（最優先）
- **絶対パスの禁止**: ソースコード、ドキュメント、スクリプト内で、絶対パス（例: `C:\Users\...`）を絶対に含めない。常にプロジェクトルートからの相対パスを使用すること。
- **機密情報のハードコード禁止**: パスワード、APIキー、認証トークン、AWSキー、秘密鍵、データベース接続文字列などを絶対に記述しない。
- **公開前セキュリティスキャンの義務化**: GitHubへのプッシュや外部配布の前に、必ず環境依存のキーワード（ユーザー名、`C:\Users` 等）でプロジェクト全件を検索し、漏洩がないか確認すること。
- **環境変数の利用**: 機密情報が必要な場合は、環境変数または `.env` ファイル（`.gitignore` で除外必須）を使用すること。
- **.gitignore の自動設定**: プロジェクトのルートに適切な `.gitignore` を作成・更新すること。特に `.gitignore` 自体に加え、`.docs/*`、`.agents/*` を必ず含め、ルールや機密情報が公開リポジトリに流出するのを防ぐこと。

# 効率的な開発プロセス
- **タスクの進捗管理**: 常に `task.md` を最新の状態に保ち、現在のステップを明確にすること。大きなタスクは小さなサブタスクに分割すること。
- **段階的な実装と検証**: 一度に多くの変更を行わず、機能単位で実装・検証を繰り返すこと。変更のたびに関連機能が破壊されていないか確認すること。
- **README の言語**: 公開用リポジトリの場合、`README.md` は原則として英語で作成すること。

# ドキュメントと同期
- **多言語同期**: 多言語でドキュメントを展開している場合、機能変更時はすべての言語版を同時に更新し、内容の矛盾を防ぐこと。
