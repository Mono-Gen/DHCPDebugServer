# コードスタイルガイドおよび開発ルール

## マニュアルの作成
- 英文 (`manual_EN.md`) および 日本語 (`manual_JP.md`) のマニュアルを、プロジェクトのルートディレクトリ（カレントディレクトリ）に作成・更新すること。
- **ドキュメントの同期**: 機能変更や使い方の更新を行う際は、両言語のマニュアルを同時に更新し、内容に矛盾がないようにすること。

## セキュリティと品質
- 安全第一、セキュリティチェック、バリデーションチェック、およびコードのリファクタリングを必ず行うこと。
- **絶対パスの禁止**: ソースコード、ドキュメント、およびスクリプト内で、絶対パス（例: `C:\Users\...`）を絶対に含めない。常にプロジェクトルートからの相対パスを使用すること。
- **環境情報の秘匿**: ユーザー名、特定のローカルディレクトリ名、PC名などの環境依存情報をコードに含めない。
- **公開前スキャンの徹底**: GitHubへのプッシュやリリース作成の前に、必ず `C:\Users` やユーザー名などのキーワードでプロジェクト全件を検索し、機密情報の漏洩がないか確認すること。
- **機密情報のハードコード禁止**: パスワード、APIキー、認証トークン、個人識別情報（ID等）、AWSアクセスキー、GitHubトークン、プライベートキー（.pem/.key）、データベース接続文字列などを絶対にソースコードに記述しない。
- **環境変数の利用**: 機密情報や環境依存の設定が必要な場合は、環境変数または `.env` ファイル（必ず `.gitignore` で除外すること）を使用すること。
- コンパイルする前に、少なくとも2回以上はコードに問題がないか、仕様を満たしているかをチェックすること。
- 回答や修正の速度よりも安定性やバグがないことを優先すること。

## 開発ルール
- **依存関係の管理**: 新しいライブラリをインストールした際は、必ず `requirements.txt` を最新の状態に更新すること。
- **Python実行**: ライブラリは `venv` で構成・実行すること。

## 指示事項
- 改善案やアイデアがあれば積極的に提案すること。
- APIキーを許可なく使用しないこと。
- フォルダやファイルを確認なく削除しないこと。
- 重要な操作（破壊的な変更、大規模なリファクタリングなど）を行う前は必ず事前に確認をとること。
- 不明点は推測で進めず、必ず質問すること。
- 誤りがあった場合は正直に認め、速やかに修正すること。

## 効率的な開発プロセス
- **タスクの進捗管理**: 常に `task.md` を最新の状態に保ち、現在のステップを明確にすること。大きなタスクは小さなサブタスクに分割して実行すること。
- **段階的な実装と検証**: 一度に多くの変更を行わず、機能単位で実装・検証を繰り返すこと。変更のたびに、関連する機能が破壊されていないか確認すること。
- **コードのモジュール化**: 1つの関数やクラスは1つの責任を持つように設計し、必要に応じてファイルを分割して再利用性を高めること。
- **自動テストの優先**: 複雑なロジックを実装する際は、可能な限りテストスクリプトを作成し、自動で正当性を検証できるようにすること。

## .gitignore
- 適切な設定を日本語で記載した`.gitignore`ファイルをカレントディレクトリに作成すること。
- `.gitignore`、`.docs/*`、`.agents/*`を含めること。
